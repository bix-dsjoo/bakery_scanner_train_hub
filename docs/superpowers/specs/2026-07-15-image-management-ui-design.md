# 이미지 관리 MVP UI 디자인

- 상태: 승인됨
- 작성일: 2026-07-15
- 대상 프로젝트: Bakery Scanner Train Hub
- 선행 문서: [브랜드별 베이커리 이미지 관리 MVP 설계](2026-07-15-bakery-image-management-design.md)
- 기술 문서: [이미지 관리 MVP 기술 설계](2026-07-15-image-management-technical-design.md)

## 1. 목적

이 문서는 이미지 관리 MVP의 시각 언어, 화면 배치, 공통 컴포넌트, 라벨링 캔버스와 접근성 기준을 정의한다. 구현 전에 최소 디자인 체계를 고정해 화면마다 색상·여백·상태 표현이 달라지는 문제를 방지한다.

화면의 목적은 브랜드와 상품을 관리하고, 많은 트레이 사진을 빠르고 정확하게 라벨링하는 것이다. 마케팅 사이트처럼 장식하기보다 실제 데이터 작업의 속도, 판독성, 오류 복구를 우선한다.

## 2. 디자인 방향

전체 방향은 다음 세 요소를 결합한다.

- Supabase Studio의 관리 도구 구조: 고정 탐색, 밀도 있는 목록, 필터와 명확한 페이지 계층
- Vercel Geist의 정밀함: 제한된 표면 단계, 얇은 테두리, hover·active·focus의 명확한 대비
- BIXOLON의 브랜드 정체성: 오렌지, Pretendard, 로고의 `X`에서 가져온 대각선

이 조합은 특정 서비스의 화면을 복제하기 위한 것이 아니다. Supabase와 Vercel의 공식 디자인 시스템에서 관리 도구에 적합한 원칙을 참고하고 BIXOLON의 색상·서체·시각 개념으로 다시 구성한다.

## 3. 브랜드 적용

2026년 4월 BIXOLON Brand Guidelines의 다음 규칙을 적용한다.

- 기본 브랜드 색상은 BIXOLON Orange `#EE7203`이다.
- 웹 서체는 Pretendard다.
- 검정·흰색·승인된 회색을 기본 표면과 본문에 사용한다.
- 대각선과 반투명 삼각형은 BIXOLON 로고의 `X`에서 유래한 시각 개념이다.
- 오렌지 tint는 배경과 그래픽 요소에 사용할 수 있지만 로고나 본문 글자에 적용하지 않는다.
- 로고 주변에는 공식 가이드가 정한 여백을 유지한다.

공식 로고 파일을 임의로 다시 그리거나 PDF에서 잘라내어 제품 자산으로 사용하지 않는다. 저장소에 공식 SVG 또는 PNG가 제공되기 전에는 앱 이름을 일반 Pretendard 텍스트로 표시하고, 제공된 이후에만 로고로 교체한다.

## 4. 시각 토큰

### 4.1 핵심 팔레트

| 역할 | 값 | 용도 |
|---|---:|---|
| Canvas | `#F5F5F5` | 앱 기본 배경 |
| Surface | `#FFFFFF` | 목록, 편집 패널, 팝오버 |
| Ink | `#000000` | 기본 글자, 대표 행동 |
| Muted Ink | `#878786` | 보조 설명, 시간, 파일 정보 |
| BIXOLON Orange | `#EE7203` | 브랜드, 선택, 포커스, 진행 상태 |
| Information Blue | `#05498D` | 정보성 강조와 링크 |

필요한 shadcn 의미 토큰은 이 팔레트에서 파생한다.

- `background`: `#F5F5F5`
- `foreground`: `#000000`
- `card`, `popover`: `#FFFFFF`
- `primary`: `#000000`
- `primary-foreground`: `#FFFFFF`
- `muted`: `#F5F5F5`
- `muted-foreground`: `#878786`
- `accent`: BIXOLON Orange를 흰색에 10% 혼합한 `#FDF1E6`
- `accent-foreground`: `#000000`
- `border`, `input`: 검정 12%의 중립 테두리 `#E0E0E0`
- `ring`: `#EE7203`
- `destructive`: 짙은 빨간색 `#B42318`

오렌지 면 위에 일반 크기의 흰색 글자를 사용하지 않는다. 대표 버튼은 검정 배경과 흰색 글자를 사용하고 오렌지는 선택, 포커스, 활성 표시와 라벨링 박스에 집중한다.

### 4.2 서체

Pretendard 한 가족만 사용해 브랜드 규정을 지키고 추가 웹 폰트 다운로드를 줄인다.

| 역할 | 크기 | 굵기 |
|---|---:|---:|
| 페이지 제목 | 24px | 700 |
| 구역 제목 | 18px | 600 |
| 본문·표 | 14px | 400 또는 500 |
| 입력 라벨 | 13px | 500 |
| 파일명·시간·보조 정보 | 12px | 400 |

수량과 진행 번호에는 `font-variant-numeric: tabular-nums`를 적용한다. 지원 굵기는 400, 500, 600, 700만 포함한다.

### 4.3 형태와 깊이

- shadcn 기본 `radius`는 6px다.
- 작은 badge와 조절점은 4px, 큰 dialog와 sheet는 8px까지 허용한다.
- 본문 구획은 Card 남용 대신 여백과 1px 구분선으로 나눈다.
- 그림자는 popover, dropdown, dialog처럼 다른 표면 위에 떠 있는 요소에만 사용한다.
- 한 화면에 넓은 오렌지 면을 반복하지 않는다.

## 5. shadcn/ui 구성

다음 설정으로 도입한다.

- 스타일: `base-nova`
- 기반 색상: `neutral`
- 접근성 primitive: Base UI
- 아이콘: Lucide
- 스타일링: Tailwind CSS 4와 CSS 변수
- React Server Components: 사용하지 않음
- 다크 모드: 사용하지 않음

도입 대상은 다음과 같다.

- 입력과 행동: `Button`, `Input`, `Select`, `Combobox`
- 탐색과 오버레이: `Sidebar`, `Sheet`, `Dialog`, `AlertDialog`, `DropdownMenu`
- 데이터와 상태: `Table`, `Tabs`, `Badge`, `Progress`, `Skeleton`
- 보조: `Tooltip`, `Sonner`, `Separator`

필요한 컴포넌트만 저장소로 가져온다. shadcn 컴포넌트는 외부 패키지의 검은 상자가 아니라 저장소가 소유하는 UI 원본으로 취급하며, 공통 변경은 `components/ui`에서 수행한다. 기능 화면이 각자 버튼이나 입력 스타일을 복제하지 않는다.

## 6. 일반 관리 화면

```text
┌──────────────┬─────────────────────────────────────────┐
│ 앱 이름      │ 브랜드명 / 상품 관리       [상품 추가] │
│ [브랜드 선택]├─────────────────────────────────────────┤
│              │ 검색 · 상태 필터 · 상품 필터           │
│ 오늘의 작업  ├─────────────────────────────────────────┤
│ 상품 관리    │ 썸네일  상품명  상태  사진 수  작업    │
│ 트레이 사진  │ 썸네일  상품명  상태  사진 수  작업    │
│              │ ...                                     │
└──────────────┴─────────────────────────────────────────┘
```

- 1280px 이상에서 224px 고정 사이드바를 사용한다.
- 사이드바 상단에는 앱 이름과 브랜드 선택기를 둔다.
- 메뉴는 `오늘의 작업`, `상품 관리`, `트레이 사진` 세 개만 둔다.
- 현재 메뉴 앞에는 짧은 오렌지 대각선 표시를 둔다.
- 본문은 페이지 제목, 대표 행동, 필터, 데이터 목록 순서를 유지한다.
- 상품과 트레이 사진은 48~64px 썸네일이 있는 목록으로 표시한다.
- 모든 화면에 하나의 대표 행동만 둔다.

`오늘의 작업`에는 라벨 필요 수, 완료 수, `이어서 라벨링하기`만 표시한다. 모델 지표, 그래프와 배포 상태를 추가하지 않는다.

## 7. 라벨링 집중 화면

```text
┌──────────────────────────────────────────────────────────────┐
│ ← 목록  IMG_0123.jpg  12 / 84       저장됨  [이 사진 완료] │
├──────────────────────────────────────────┬───────────────────┤
│                                          │ 선택한 박스       │
│                                          │ 상품 검색         │
│        어두운 이미지 캔버스              │ 상품 지정         │
│        박스 생성·이동·크기 변경          │                   │
│                                          │ 단축키 안내       │
│                                          │ 박스 목록         │
├──────────────────────────────────────────┴───────────────────┤
│ N 새 박스   Delete 삭제   Esc 취소   Enter 완료             │
└──────────────────────────────────────────────────────────────┘
```

- 라벨링 화면에서는 사이드바와 관리 메뉴를 숨긴다.
- 상단에는 목록 이동, 파일명, 진행 순서, 저장 상태, 완료 버튼만 둔다.
- 이미지 바깥 캔버스는 짙은 중립색 `#1F1F1F`을 사용하되 앱 테마 자체는 라이트로 유지한다.
- 오른쪽 320px 패널은 선택한 박스와 상품 지정만 담당한다.
- 화면이 좁아지면 편집 패널을 shadcn `Sheet`로 전환한다.
- 캔버스에는 브랜드 대각선과 삼각형 장식을 사용하지 않는다.

### 7.1 박스 표현

- 지정 완료 박스: 흰색 2px 선과 어두운 외곽 그림자
- 선택 박스: BIXOLON Orange 3px 선, 어두운 외곽선, 10px 조절점
- 상품 미지정 박스: BIXOLON Yellow `#FAB600` 2px 점선과 `상품 미지정` 문구
- 선택 박스는 겹친 박스보다 위에 표시
- 상품명은 박스 위 작은 라벨에 표시
- 긴 상품명은 줄임표로 줄이고 tooltip에서 전체 내용 제공

색만으로 상태를 구분하지 않는다. 선 종류, 굵기, label 문구를 함께 사용한다.

## 8. 컴포넌트 밀도와 상태

- 일반 입력과 버튼 높이: 36px
- 대표 행동 버튼 높이: 40px
- 목록 행 높이: 52~64px
- 아이콘 기본 크기: 16px
- 데스크톱의 최소 상호작용 영역: 32px

버튼 규칙은 다음과 같다.

- 대표 행동: 검정 배경, 흰색 글자
- 보조 행동: 흰색 배경, 중립 테두리
- 선택과 활성: 오렌지 선, 배경 tint, 검정 글자
- 위험 행동: 빨간색과 명시적인 `삭제` 문구
- icon-only 버튼: 접근 가능한 이름과 tooltip 필수

상태는 `저장 중`, `저장됨`, `저장하지 못함`, `라벨 필요`, `완료`처럼 제품 용어를 그대로 표시한다. 내부 상태값이나 구현 용어를 노출하지 않는다.

## 9. 움직임

- hover, dropdown, sheet 전환은 120~180ms다.
- page 진입과 목록 행에 장식적인 등장 애니메이션을 사용하지 않는다.
- 애니메이션 속성은 `opacity`와 `transform`처럼 의도한 항목만 지정한다.
- `prefers-reduced-motion`에서는 필수적이지 않은 전환을 제거한다.
- 자동 저장 중에도 캔버스 조작을 막지 않는다.

## 10. 빈 화면, 로딩과 오류

빈 화면은 원인과 다음 행동 하나만 제공한다.

- 상품 없음: `첫 상품을 등록해 주세요`와 `상품 등록`
- 트레이 사진 없음: `라벨링할 트레이 사진이 없어요`와 `사진 올리기`
- 검색 결과 없음: `조건에 맞는 사진이 없어요`와 `필터 초기화`
- 모든 작업 완료: `라벨링을 모두 마쳤어요`

빈 화면에서만 작은 BIXOLON `X` 대각선 패턴을 사용할 수 있다.

로딩과 오류는 다음 규칙을 따른다.

- Skeleton은 최종 내용과 같은 크기로 표시한다.
- 200ms보다 짧은 요청에는 loading 표시를 생략한다.
- 입력 오류는 해당 입력 아래에 표시한다.
- 업로드 오류는 파일별 원인과 조치를 목록으로 표시한다.
- 자동 저장 오류는 저장 상태 영역에 유지하고 `다시 시도`를 제공한다.
- Sonner toast는 업로드 완료처럼 화면 밖에서도 알아야 하는 결과에만 사용한다.
- 오류 문구는 실패 원인과 사용자가 취할 다음 행동을 함께 설명한다.

## 11. 반응형 범위

- 1280px 이상: 고정 사이드바와 전체 정보 표시
- 1024~1279px: 축소 탐색과 축소된 라벨링 편집 패널
- 1024px 미만: 관리 화면 사용 가능, 라벨링에는 데스크톱 사용 안내 표시
- 모바일 전용 라벨링 UI: MVP 범위 밖

관리 화면에서 사이드바가 들어갈 공간이 없으면 shadcn `Sheet`로 전환한다. 브라우저 확대 200%에서도 대표 행동과 저장 상태가 가려지지 않아야 한다.

## 12. 접근성

- 모든 일반 관리 기능과 라벨링 단축키를 키보드로 사용할 수 있어야 한다.
- focus-visible은 2px BIXOLON Orange ring과 바깥 여백으로 표시한다.
- 상태는 색상, 아이콘, 선 모양, 문구를 함께 사용한다.
- 저장과 업로드 결과는 적절한 `aria-live` 영역에서 알린다.
- dialog와 sheet는 focus trap, 초기 focus, 닫은 뒤 focus 복원을 지원한다.
- 브라우저 확대를 제한하지 않는다.
- 긴 사용자 입력과 번역된 문구에서도 layout이 깨지지 않아야 한다.
- 명암 대비는 자동 검사와 실제 브라우저 검사를 모두 통과해야 한다.

## 13. 테스트와 시각 검증

### 13.1 자동 테스트

- Vitest와 React Testing Library: component variant, 상태 문구, keyboard focus
- Playwright: 주요 흐름, 화면 크기, 저장 상태, 캔버스 상호작용
- `axe-core`: 이름 없는 버튼, label 연결, 주요 ARIA와 명암 대비 오류

### 13.2 시각 검증 상태

다음 상태를 1440×900, 1280×720, 1024×768에서 확인한다.

- 빈 목록, loading, 오류, 100개 이상 목록
- 긴 한글 상품명과 긴 파일명
- 밝은 사진과 어두운 사진 위의 박스
- 저장 중, 저장됨, 저장 실패
- 지정 완료, 상품 미지정, 선택된 박스
- keyboard focus와 브라우저 확대 200%

스크린샷 검증은 정확한 pixel 동일성보다 layout 잘림, 겹침, 의도하지 않은 줄바꿈, 상태 누락을 우선한다. 최종 확인은 Windows의 실제 Chrome 또는 Edge에서도 수행한다.

## 14. 비목표

- 다크 모드
- 브랜드별 사용자 선택 테마
- 모바일 전용 라벨링
- 마케팅용 hero와 장식 애니메이션
- 모든 화면을 Card로 구성하는 대시보드
- 제품과 관계없는 gradient, glow, glassmorphism

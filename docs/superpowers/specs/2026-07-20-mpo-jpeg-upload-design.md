# JPEG 기반 MPO 업로드 호환 설계

- 상태: 승인됨
- 작성일: 2026-07-20
- 선행 문서: [이미지 관리 MVP 설계](2026-07-15-bakery-image-management-design.md), [이미지 관리 MVP 기술 설계](2026-07-15-image-management-technical-design.md)

## 1. 배경

일부 iPhone JPEG 사진은 주 이미지와 보조 이미지를 하나의 파일에 담는 MPO(Multi-Picture Object) 구조를 사용한다. 파일명과 JPEG 시그니처는 정상이고 일반 이미지 뷰어에서도 열리지만, Pillow는 이를 `JPEG`가 아닌 `MPO`로 판별한다. 현재 이미지 검증기는 Pillow의 포맷명이 `JPEG`, `PNG`, `WEBP`인 경우만 허용하므로 이런 사진을 `IMAGE_UNSUPPORTED`로 거부한다.

실제 재현 파일은 첫 번째 RGB 프레임과 두 번째 흑백 보조 프레임을 포함하며 두 프레임 모두 정상적으로 디코딩된다. 같은 촬영 묶음의 단일 프레임 JPEG는 기존 검증을 통과한다.

## 2. 결정

사용자가 `.jpg` 또는 `.jpeg` 파일로 선택한 JPEG 기반 MPO를 기존 JPEG 업로드의 호환 형태로 허용한다.

- 원본 MPO 파일은 재인코딩하지 않고 그대로 보존한다.
- SHA-256은 사용자가 올린 원본 바이트를 기준으로 계산한다.
- 이미지 크기와 썸네일은 MPO의 첫 번째 프레임을 기준으로 한다.
- 저장 메타데이터는 기존 계약과 동일하게 `mime_type=image/jpeg`, 저장 확장자 `jpg`를 사용한다.
- `.mpo` 확장자를 새 사용자 지원 형식으로 추가하지 않는다.
- PNG, WebP와 일반 JPEG의 기존 동작은 변경하지 않는다.

이 결정은 사용자가 JPEG로 인식하는 사진을 별도 변환 없이 등록하게 하면서, 원본을 변경하지 않는 저장 원칙과 기존 API 계약을 유지한다.

## 3. 검증과 처리 흐름

이미지 검증기는 Pillow가 보고한 실제 포맷을 다음과 같이 정규화한다.

| Pillow 포맷 | 허용 파일 확장자 | 저장 MIME | 저장 확장자 | 처리 대상 |
|---|---|---|---|---|
| `JPEG` | `.jpg`, `.jpeg` | `image/jpeg` | `jpg` | 단일 이미지 |
| `MPO` | `.jpg`, `.jpeg` | `image/jpeg` | `jpg` | 첫 번째 프레임 |
| `PNG` | `.png` | `image/png` | `png` | 단일 이미지 |
| `WEBP` | `.webp` | `image/webp` | `webp` | 단일 이미지 |

검사 순서는 다음과 같다.

1. Pillow로 파일을 열고 포맷을 확인한다.
2. 전체 파일 구조를 `verify()`로 검증한다.
3. 다시 열어 첫 번째 프레임을 명시적으로 선택하고 완전히 디코딩한다.
4. 첫 번째 프레임의 크기를 이미지 메타데이터로 사용한다.
5. 실제 포맷과 사용자 파일 확장자의 대응을 검증한다.
6. MPO는 JPEG 메타데이터로 정규화해 이후 저장 흐름에 전달한다.

썸네일 생성도 원본을 다시 열고 첫 번째 프레임을 명시적으로 선택한 뒤 EXIF orientation을 적용한다. 보조 프레임은 썸네일이나 학습용 이미지로 사용하지 않는다.

## 4. 오류 처리

- 정상적으로 열리지만 허용 목록에 없는 형식은 기존처럼 `IMAGE_UNSUPPORTED`를 반환한다.
- MPO 구조가 손상되었거나 첫 번째 프레임을 완전히 디코딩할 수 없으면 `IMAGE_CORRUPT`를 반환한다.
- MPO 내용과 `.jpg`·`.jpeg` 이외의 파일 확장자가 조합되면 확장자 위장으로 보고 `IMAGE_UNSUPPORTED`를 반환한다.
- 기존 용량, 디스크 여유 공간, 브랜드별 SHA-256 중복 규칙은 그대로 적용한다.

## 5. 테스트

실제 업무 파일이나 네트워크 공유에 의존하지 않는 작은 합성 MPO를 Pillow의 `save_all`과 두 개의 프레임으로 테스트 중 생성한다.

- `.jpg` 이름의 MPO 검사 결과가 `image/jpeg`, `jpg`, 첫 프레임 크기인지 검증한다.
- MPO 썸네일이 첫 프레임의 색상·크기와 EXIF orientation을 사용하는지 검증한다.
- `.mpo` 이름의 같은 내용은 계속 거부되는지 검증한다.
- 손상된 MPO는 읽을 수 없는 이미지로 거부되는지 검증한다.
- 기존 JPEG, PNG, WebP와 확장자 위장 테스트를 회귀 실행한다.
- 업로드 사용 사례에서 MPO가 `COMPLETED` 또는 `UNLABELED`의 기존 종류별 상태로 저장되는지 확인한다.

## 6. 비목표

- 보조 프레임을 별도 파일이나 데이터베이스 레코드로 추출하지 않는다.
- HDR gain map, depth map 등 보조 프레임의 의미를 해석하지 않는다.
- `.mpo`를 별도의 사용자 지원 형식이나 MIME 계약으로 노출하지 않는다.
- 기존 원본을 일반 JPEG로 재인코딩하거나 보조 프레임을 삭제하지 않는다.

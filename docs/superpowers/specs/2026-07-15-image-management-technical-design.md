# 이미지 관리 MVP 기술 설계

- 상태: 승인됨
- 작성일: 2026-07-15
- 대상 프로젝트: Bakery Scanner Train Hub
- 선행 문서: [브랜드별 베이커리 이미지 관리 MVP 설계](2026-07-15-bakery-image-management-design.md)

## 1. 목적과 범위

이 문서는 승인된 이미지 관리 MVP를 Windows PC 한 대에서 실제 업무 데이터로 운영하기 위한 기술 구성을 정의한다. 최대 이미지 수는 10만 장이며, 주 사용자는 한 명이다. 애플리케이션은 회사 내부망의 다른 PC에서도 웹 브라우저로 접속할 수 있다.

포함 범위는 브랜드와 상품 관리, 상품 사진과 트레이 사진 업로드, 박스 라벨링, 검색과 필터, 자동 저장이다. 모델 학습, 데이터셋 분할과 내보내기, 모델 관리, 외부 인터넷 공개, 로그인과 권한 관리는 포함하지 않는다.

## 2. 결정 요약

### 2.1 기술 스택

- 프런트엔드: React 19.2, TypeScript, Vite 8, Tailwind CSS 4
- UI 컴포넌트: shadcn/ui `base-nova`, Radix UI, Lucide
- 라벨링 캔버스: Konva 10과 `react-konva`
- 백엔드: Python 3.13, FastAPI
- 데이터 접근: SQLAlchemy 2.0, Alembic
- 데이터베이스: SQLite WAL
- 이미지 처리: Pillow
- Python 환경과 잠금 파일: `uv`, `uv.lock`
- Node.js 런타임: Node.js 24 LTS, npm 잠금 파일
- 백엔드 테스트: pytest
- 프런트엔드 테스트: Vitest, React Testing Library, Playwright, axe-core

Python은 최신 버전 자체보다 이후 ML 코드와의 보수적인 생태계 호환성을 우선해 3.13을 사용한다. Python과 JavaScript 의존성의 정확한 패치 버전은 각각 `uv.lock`과 `package-lock.json`에 고정한다. FastAPI는 0.x 버전 정책에 맞춰 직접 의존성의 호환 minor 범위를 선언하고 잠금 파일로 실제 설치 버전을 고정한다.

### 2.2 선택 이유

- Python 중심 팀이 FastAPI의 업무 규칙과 이후 ML 연동 코드를 함께 유지할 수 있다.
- React와 Konva는 박스 생성·이동·크기 변경이 많은 상호작용을 브라우저에서 구현하기 적합하다.
- 서버 렌더링이 필요 없는 사내 도구이므로 Next.js 대신 Vite SPA를 사용한다.
- 단일 사용자, 단일 서버 프로세스, 10만 장 이하 조건에서는 SQLite와 로컬 파일 저장소가 충분하다.
- PostgreSQL, Redis, S3 호환 저장소, 작업 큐, Docker는 MVP에 도입하지 않는다.

## 3. 배포와 실행 구조

MVP는 하나의 저장소와 하나의 운영 서버 프로세스로 구성한다.

```text
브라우저
  └─ HTTP
      └─ FastAPI 단일 프로세스
          ├─ /api/v1/*         업무 API
          ├─ /assets/*         빌드된 React 정적 파일
          ├─ SQLite            메타데이터와 박스
          └─ 로컬 파일 저장소  원본과 썸네일
```

개발 중에는 Vite 개발 서버가 `/api` 요청을 FastAPI로 프록시한다. 실제 사용 시에는 Vite가 만든 `dist`를 FastAPI가 제공하므로 CORS 설정이나 별도 프런트엔드 서버가 필요 없다.

운영 서버는 `0.0.0.0:8000`을 기본 주소로 사용한다. 같은 PC에서는 `http://localhost:8000`, 사내망에서는 `http://<서버-PC-IP>:8000`으로 접속한다. 호스트와 포트는 환경 설정으로 바꿀 수 있다.

루트의 `start.ps1`은 다음 순서로 실행한다.

1. Python 3.13, `uv`, Node.js 24가 준비되었는지 검사한다.
2. 잠금 파일을 기준으로 누락된 의존성을 준비한다.
3. React 운영 빌드를 생성한다.
4. Alembic 마이그레이션을 최신 상태로 적용한다.
5. FastAPI를 Uvicorn 단일 worker로 시작한다.

Python 3.13, `uv`, Node.js 24는 PC에 한 번 설치해야 하는 시스템 선행 조건이다. 이 조건을 갖춘 뒤에는 초기 프로젝트 의존성 설치와 일상 실행을 모두 `start.ps1` 하나로 수행한다. 선행 도구가 없거나 의존성 파일을 내려받을 수 없으면 누락 항목과 조치 방법을 표시하고 종료한다.

## 4. 저장소 구조

```text
backend/
  app/
    api/             HTTP 요청·응답과 라우팅
    application/     업로드, 라벨 저장, 완료 처리 사용 사례
    domain/          데이터 규칙과 상태 전이
    infrastructure/  SQLite, 파일 저장소, 이미지 처리
  migrations/        Alembic 마이그레이션
  tests/
frontend/
  src/
    app/             라우팅과 전역 브랜드 맥락
    components/ui/   저장소가 소유하는 shadcn 공통 컴포넌트
    pages/           한 화면의 대표 작업
    features/        상품, 업로드, 검색, 라벨링
    shared/          공통 UI와 API 클라이언트
  tests/
docs/
start.ps1
pyproject.toml
uv.lock
```

프런트엔드는 서버 상태, 현재 브랜드 맥락, 라벨링 편집 중 상태를 분리한다. 박스 좌표 계산과 검증은 캔버스 컴포넌트 밖의 순수 함수로 유지한다. 백엔드는 API 계층이 SQLAlchemy나 파일 경로를 직접 다루지 않고 application 계층의 사용 사례를 호출한다.

## 5. 업무 구성요소

### 5.1 프런트엔드

- 브랜드 선택기: 현재 브랜드를 모든 주요 화면의 조회 범위로 유지한다.
- 오늘의 작업: 라벨 필요·완료 수와 `이어서 라벨링하기`를 제공한다.
- 상품 관리: 상품 등록·수정·비활성화와 상품 사진 관리를 담당한다.
- 트레이 사진 관리: 일괄 업로드, 검색, 필터, 페이지 조회를 담당한다.
- 라벨링 편집기: 박스 편집, 상품 지정, 자동 저장, 완료와 다음 사진 이동을 담당한다.
- 공통 UI: 버튼, 검색, 필터, 상태 표시, 오류 메시지의 공통 동작을 제공한다.

### 5.2 백엔드

- 브랜드·상품 서비스: 고유성, 활성 상태, 브랜드 격리 규칙을 적용한다.
- 이미지 등록 서비스: 스트리밍 수신, 검증, 해시, 중복 검사, 썸네일, 저장을 조정한다.
- 이미지 조회 서비스: 브랜드별 검색·필터·페이지 조회를 제공한다.
- 라벨링 서비스: 박스 검증, 전체 교체 저장, 상태 전이, revision 충돌 검사를 담당한다.
- 파일 저장소: 저장 키와 실제 Windows 경로의 변환, 원본·썸네일·임시 파일 수명 주기를 담당한다.
- 데이터베이스: SQLite 연결, 트랜잭션, 인덱스, 마이그레이션을 담당한다.

## 6. 로컬 데이터 저장

애플리케이션 소스와 업무 데이터를 분리한다. 기본 데이터 루트는 `C:\BakeryScannerData`이며 환경 변수 `BAKERY_DATA_DIR`로 바꿀 수 있다. 업무 데이터는 Git에 추가하지 않는다.

```text
C:\BakeryScannerData\
  database\app.db
  originals\<brand-id>\<hash-앞2자리>\<다음2자리>\<무작위키>.<확장자>
  thumbnails\<brand-id>\<hash-앞2자리>\<다음2자리>\<무작위키>.webp
  imports\
  trash\
  logs\app.log
```

원본은 다시 인코딩하지 않고 검증된 파일을 그대로 보존한다. 썸네일은 가로·세로 중 긴 변이 최대 512픽셀이 되도록 축소한 WebP로 만든다. 파일명은 사용자가 전달한 값을 경로로 사용하지 않고 서버가 생성한 저장 키만 사용한다.

파일을 해시 앞 네 자리 기준의 두 단계 폴더로 분산해 폴더 하나에 파일이 몰리지 않게 한다. 같은 이미지가 다른 브랜드에 등록될 수 있으므로 브랜드 폴더와 무작위 저장 키를 포함하고, 브랜드별 파일 수명 주기를 독립적으로 유지한다.

자동 백업과 복구 기능은 MVP에 포함하지 않는다. 앱 재시작 후에는 데이터가 유지되지만 디스크나 PC가 고장 나면 복구할 수 없다. 이 위험을 README와 운영 안내에 명시한다.

## 7. 데이터베이스

SQLite는 다음 연결 정책을 사용한다.

- WAL journal mode
- foreign key enforcement 활성화
- busy timeout 설정
- 애플리케이션 서버 worker 한 개
- 쓰기 작업별 짧은 트랜잭션

핵심 테이블은 제품 설계의 `Brand`, `Product`, `Image`, `BoundingBox`를 그대로 구현한다. 다음 제약과 인덱스는 DB에서도 보장한다.

- 브랜드 이름 unique
- `(brand_id, product.code)` unique
- `(brand_id, image.sha256)` unique
- 이미지 목록용 `(brand_id, kind, labeling_status, created_at, id)` index
- 박스 조회용 `(image_id)` index
- 외래 키 삭제 정책은 명시적으로 설정하고 애플리케이션 삭제 규칙과 일치시킨다.

목록은 `created_at DESC, id DESC` 순서의 cursor 기반 페이지 조회를 사용한다. 기본 페이지 크기는 50개, 최대 100개다. 검색·필터 결과도 전체 행이나 원본 이미지를 메모리에 올리지 않는다.

## 8. API 경계

모든 업무 API는 `/api/v1` 아래에 둔다.

- 브랜드: 목록, 생성, 수정, 비활성화
- 상품: 목록, 생성, 수정, 비활성화
- 이미지: 단일 파일 등록, 목록, 상세, 원본·썸네일 조회, 삭제
- 라벨: 박스 전체 저장, 사진 완료
- 작업 요약: 브랜드별 라벨 필요·완료 개수와 다음 작업 대상

브라우저의 다중 파일 선택은 파일별 API 요청으로 나누고 동시 요청 수를 2개로 제한한다. 서버는 파일 하나를 메모리에 모두 읽지 않고 임시 파일로 스트리밍하면서 용량과 SHA-256을 계산한다. 파일별 결과를 모아 부분 성공 요약을 표시하며, 한 요청 실패가 다른 파일을 되돌리지 않는다.

API 오류 응답은 다음 정보를 일관되게 제공한다.

- 기계 판별용 `code`
- 사용자 표시용 한국어 `message`
- 가능한 경우 다음 행동을 설명하는 `action`
- 필드 검증 실패일 때 `field_errors`

존재하지 않는 자원은 404, 중복과 revision 충돌은 409로 반환한다. 입력 형식 오류와 업무 규칙 오류는 서로 구분되는 오류 코드를 사용한다.

## 9. 업로드 데이터 흐름

### 9.1 공통 처리

1. 브라우저가 현재 브랜드와 이미지 종류를 화면 맥락에서 확정한다.
2. 파일을 `imports`에 스트리밍하며 최대 25MB를 강제하고 SHA-256을 계산한다.
3. JPEG, PNG, WebP 여부와 이미지 디코딩 가능 여부를 실제 내용으로 확인한다.
4. 같은 브랜드의 SHA-256 중복을 검사한다.
5. 썸네일을 임시 위치에 생성한다.
6. 원본과 썸네일을 최종 저장 키로 이동한다.
7. 이미지 메타데이터를 DB 트랜잭션으로 저장한다.
8. 성공 결과와 생성된 이미지 ID를 반환한다.

단계가 실패하면 해당 요청이 만든 임시 파일과 최종 파일을 보상 정리한다. 서버 시작 시 24시간 이상 지난 `imports` 파일을 정리한다. 업로드 전 현재 요청 처리 후 최소 10GB의 여유 공간을 남길 수 있는지 검사하고, 그렇지 않으면 파일을 받기 전에 차단한다.

### 9.2 상품 사진

상품 상세 화면이 `kind=PRODUCT`와 현재 `product_id`를 자동으로 전달한다. 서버는 상품이 현재 브랜드의 활성 상품인지 확인한다. 정상 등록된 상품 사진은 즉시 `COMPLETED`다.

### 9.3 트레이 사진

트레이 사진 화면이 `kind=TRAY`를 자동으로 전달하며 이미지의 `product_id`는 비어 있다. 정상 등록된 트레이 사진은 `UNLABELED`다.

## 10. 라벨링 데이터 흐름

1. 편집기는 원본 이미지, 현재 박스 전체, `revision`을 함께 읽는다.
2. 박스 생성·이동·크기 변경·삭제를 UI에 즉시 반영한다.
3. 좌표를 원본 기준 0 이상 1 이하의 `x1`, `y1`, `x2`, `y2`로 변환한다.
4. 마지막 변경 후 500ms가 지나면 예상 `revision`과 현재 박스 전체를 저장한다.
5. 서버는 이미지 종류, 좌표, 상품 브랜드, 상품 활성 상태와 `revision`을 검증한다.
6. 하나의 트랜잭션에서 기존 박스를 요청의 전체 목록으로 교체하고 `revision`을 1 증가시킨다.
7. 성공한 새 `revision`과 이미지 상태를 응답한다.

전체 교체 방식은 한 이미지의 박스 수가 제한적이라는 전제에서 생성·수정·삭제 API를 따로 조합할 때 생기는 부분 저장을 방지한다. 원본 이미지 파일은 라벨 저장 시 변경하지 않는다.

완료 요청에는 예상 `revision`을 포함한다. 박스가 한 개 이상이고 모든 박스의 좌표와 상품이 유효할 때만 `COMPLETED`로 전환한다. 완료된 사진에 미지정 박스가 생기거나 모든 박스가 삭제되면 `UNLABELED`로 돌아간다.

## 11. 오류 처리와 일관성

### 11.1 자동 저장

UI는 `저장 중`, `저장됨`, `저장하지 못함`을 표시한다. 네트워크 오류가 발생하면 편집 중인 박스를 메모리에 유지하고 사용자가 재시도할 수 있게 한다. 미저장 변경이 남은 상태에서 화면 이동이나 새로고침을 시도하면 경고한다.

서버 `revision`이 예상 값과 다르면 409를 반환한다. 자동 병합하지 않고 충돌한 사진과 최신 내용을 다시 불러오는 행동을 안내한다.

### 11.2 파일과 DB

- DB 등록 실패 시 해당 요청이 만든 원본과 썸네일을 제거한다.
- 이미지 삭제 전에 원본과 썸네일을 같은 볼륨의 `trash`로 원자적으로 이동한다. 파일 이동 중 실패하면 이미 옮긴 파일을 원래 위치로 복원하고 DB를 변경하지 않는다.
- 파일 이동 후 연결 박스와 DB 기록을 트랜잭션으로 삭제한다. DB 삭제가 실패하면 파일을 원래 위치로 복원한다.
- DB 삭제가 끝나면 `trash`의 파일을 제거한다. 이 제거가 실패한 경우 오류를 기록하고 서버 시작 시 24시간 이상 지난 `trash` 파일을 정리한다.
- 브랜드나 상품이 사용 중이면 물리 삭제 대신 비활성화한다.

### 11.3 로그

앱 시작·종료, 업로드 실패, revision 충돌, DB 오류, 파일 정리 실패를 기록한다. 원본 이미지 내용과 불필요한 전체 사용자 경로는 기록하지 않는다. 로그는 파일당 10MB, 최대 5개로 순환한다.

## 12. 사내망 보안

- MVP에는 로그인과 사용자별 권한이 없다.
- Windows 방화벽은 한 번의 운영 설정으로 도메인·개인 네트워크 프로필에서만 서버 포트를 허용한다. `start.ps1`은 관리자 권한을 요구하거나 방화벽을 자동 변경하지 않는다.
- 공용 네트워크 프로필과 외부 인터넷 포트 포워딩은 허용하지 않는다.
- API는 모든 입력, ID 연결, 브랜드 범위, 파일 내용을 서버에서 다시 검증한다.
- 사용자는 실제 저장 경로를 API에 전달할 수 없고 이미지 ID로만 파일을 요청한다.
- 같은 origin에서 정적 UI와 API를 제공해 운영 CORS를 열지 않는다.

내부망의 다른 사용자가 주소를 알면 별도 인증 없이 데이터에 접근할 수 있다. 접근 통제가 필요해지면 외부 공개가 아니라 로그인·권한 설계를 별도 범위로 진행한다.

## 13. 테스트 전략

### 13.1 백엔드 단위 테스트

- 상품 사진의 필수 상품 연결과 완료 상태
- 트레이 사진 완료 조건과 상태 전이
- 박스 좌표 범위와 순서
- 브랜드 간 상품 연결 차단
- 비활성 상품의 신규 지정 차단
- 중복 이미지 검사
- revision 충돌

### 13.2 백엔드 통합 테스트

임시 SQLite DB와 임시 파일 루트를 사용한다.

- 정상·손상·초과 용량·중복 이미지 업로드
- 여러 파일의 부분 성공
- 원본 저장과 썸네일 생성
- DB 실패 후 파일 보상 정리
- 이미지 삭제 후 DB와 파일 정리
- WAL, foreign key, Alembic 마이그레이션
- 검색·필터·cursor 페이지 조회

### 13.3 프런트엔드 테스트

- 브랜드 맥락과 다른 브랜드 선택지 차단
- 업로드 진행과 파일별 결과
- 저장 상태와 이동 경고
- 좌표 정규화와 역변환 순수 함수
- 마우스와 키보드 박스 작업
- 충돌과 네트워크 복구 안내

### 13.4 브라우저 전체 흐름

Playwright로 브랜드 생성, 상품 등록, 상품 사진 등록, 트레이 사진 등록, 박스와 상품 지정, 자동 저장, 새로고침, 완료, 다음 사진 이동을 검증한다.

### 13.5 용량과 Windows 운영 검증

- 이미지 메타데이터 10만 건에서 검색·필터·첫 페이지 조회가 2초 안에 끝나는지 기준 PC에서 측정한다.
- 목록이 현재 페이지의 썸네일만 요청하고 원본을 내려받지 않는지 확인한다.
- 100장 일괄 등록에서 동시 요청이 2개 이하이며 메모리 사용량이 파일 수에 따라 계속 증가하지 않는지 확인한다.
- `start.ps1` 실행 후 같은 PC와 사내망의 다른 PC에서 접속되는지 확인한다.
- 서버 재시작 후 이미지, 박스, 상태가 유지되는지 확인한다.

## 14. 운영 제약과 위험

- 실제 업무 데이터는 Windows PC 한 대와 1TB 이상 디스크에 저장한다.
- 동시에 실행하는 애플리케이션 서버는 하나이며 Uvicorn worker도 하나다.
- 10만 장을 넘기거나 다수 사용자가 동시에 작업하게 되면 SQLite와 로컬 파일 저장소의 적합성을 다시 검토한다.
- 자동 백업이 없으므로 하드웨어 장애, 랜섬웨어, 사용자 실수로 인한 영구 유실 위험을 수용한다.
- 외부 인터넷 공개, HTTPS 종단, 계정과 권한은 이 설계의 운영 범위가 아니다.

- 10만 건 용량 검증과 100장 등록 검증을 통과한다.

- 같은 PC와 회사 내부망 브라우저에서 사용할 수 있다.
- 재시작 후 데이터가 유지되고 저장 실패나 충돌 시 사용자가 다음 행동을 알 수 있다.

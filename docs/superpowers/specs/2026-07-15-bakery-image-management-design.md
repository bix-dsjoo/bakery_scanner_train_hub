# 브랜드별 베이커리 이미지 관리 MVP 설계

- 상태: 승인됨
- 작성일: 2026-07-15
- 대상 프로젝트: Bakery Scanner Train Hub

## 1. 배경

Bakery Scanner Train Hub는 브랜드별 빵 이미지를 축적하고, 장기적으로 학습 데이터셋과 학습 모델의 생명주기를 관리하기 위한 프로젝트다.

첫 버전은 범위를 이미지 관리로 제한한다. 모델 학습 구조를 먼저 고정하지 않고, 향후 2단계 인식 파이프라인에 사용할 수 있는 원본 이미지와 정답 데이터를 정확하게 축적하는 것이 목적이다.

예정된 2단계 인식 파이프라인은 다음과 같다.

1. 탐지기가 이미지에서 모든 빵을 `bread` 단일 클래스로 찾는다.
2. 분류기가 탐지된 빵 이미지를 브랜드별 상품으로 분류한다.

## 2. 사용자

초기 사용자는 데이터 작업자와 ML 엔지니어다.

- 데이터 작업자는 브랜드와 상품을 등록하고, 이미지를 올리고, 트레이 이미지의 빵마다 박스와 상품을 지정한다.
- ML 엔지니어는 축적된 데이터의 상태와 구조를 확인한다.

MVP는 내부망에서 로그인 없이 사용한다. 사용자별 권한, 작업자 할당, 검수자 승인, 감사 로그는 제공하지 않는다.

## 3. 목표

- 브랜드별 상품과 이미지를 분리해 관리한다.
- 분류기 학습용 단일 상품 사진을 상품에 연결한다.
- 실제 스캐너 입력과 같은 트레이 사진에서 모든 빵을 박스로 표시하고 상품을 지정한다.
- 미라벨링 이미지와 완료 이미지를 빠르게 구분한다.
- 반복 라벨링 단계와 불필요한 선택을 최소화한다.
- 자동 저장과 충돌 감지로 라벨 유실과 덮어쓰기를 방지한다.
- 향후 탐지 데이터와 분류 데이터로 변환할 수 있는 모델 중립적 원본 데이터를 보존한다.

## 4. 비목표

다음 항목은 MVP에 포함하지 않는다.

- 모델 학습과 실험 비교
- 모델 등록, 승인, 변환, 배포
- 학습 데이터셋 버전과 train/validation/test 분할
- YOLO 또는 COCO 형식 내보내기
- 스캐너 단말의 이미지 자동 수집
- 외부 라벨링 도구 연동
- 바운딩 박스 외 세그멘테이션 라벨
- 사용자 로그인과 역할별 권한
- 별도 검수·승인 단계
- 작업 큐, 마이크로서비스, 분산 처리

## 5. 연구에 따른 모델 방향

### 5.1 검토 결과

베이커리와 자동 계산대 연구에서는 다중 클래스 탐지기 하나로 위치와 상품을 동시에 예측하는 방식이 유효한 기준선임을 확인했다.

- Semmeldetector는 18종의 베이커리 상품과 1,151개 이미지로 YOLOv8 탐지기를 학습해 AP@0.5 89.1%를 보고했다.
- 2024년 RPC 기반 연구는 YOLOv8과 YOLOv9를 미세조정해 평균 checkout accuracy 95.16%를 보고했다.

그러나 자동 계산대 문제는 상품 간 외형 차이가 작고, 상품별 데이터가 적으며, 단일 상품 촬영 환경과 실제 트레이 촬영 환경 사이에 차이가 있다는 특성이 있다. 이에 대응해 상품 후보 영역을 먼저 찾고 별도 prototype classifier, 임베딩 또는 재정렬 단계로 세부 상품을 판별하는 연구도 이어지고 있다.

### 5.2 프로젝트 결정

초기 모델 방향은 다음과 같이 정한다.

- 탐지기: 모든 빵을 `bread` 단일 클래스로 탐지
- 분류기: 탐지된 박스 영역을 브랜드별 상품으로 분류

이미지 관리 시스템은 이 모델 구조에 필요한 정답을 보관하되, 특정 학습 프레임워크나 파일 형식에 종속되지 않는다.

- 탐지 학습에서는 모든 박스를 `bread` 클래스로 변환한다.
- 분류 학습에서는 상품 사진과 상품 ID가 지정된 박스 영역을 상품별 입력 이미지로 사용한다.
- 박스에서 잘라낸 이미지는 원본 데이터로 중복 보관하지 않고 데이터셋 생성 시 파생한다.

## 6. 제품 설계 원칙

토스의 공식 디자인·개발 블로그에서 공개한 제품 원칙을 내부 라벨링 도구의 맥락에 맞게 적용한다.

### 6.1 Easy to answer

사용자가 기술적 선택을 해석하도록 요구하지 않는다. 현재 화면의 맥락으로 브랜드, 이미지 종류, 상품을 가능한 범위에서 미리 결정한다.

- 상품 상세 화면에서 사진을 추가하면 이미지 종류는 `상품 사진`, 상품은 현재 상품으로 자동 결정한다.
- 트레이 사진 화면에서 사진을 추가하면 이미지 종류는 `트레이 사진`으로 자동 결정한다.
- 상품 선택 목록에는 현재 브랜드의 활성 상품만 표시한다.

### 6.2 One thing per one page

한 화면의 대표 작업은 하나로 제한한다.

- 오늘의 작업: 다음 라벨링 작업 시작
- 상품 상세: 상품 정보와 상품 사진 관리
- 트레이 사진: 업로드와 작업 대상 탐색
- 라벨링: 박스 생성과 상품 지정

### 6.3 반복 업무 제거

- 변경 사항 자동 저장
- 최근 선택 상품 유지
- 완료 후 다음 미라벨링 이미지로 자동 이동
- 최소한의 키보드 단축키 제공
- 브랜드 선택을 화면 전환 후에도 유지

### 6.4 명확하고 간결한 문구

`어노테이션`, `객체 탐지용 이미지`, `레퍼런스 이미지`처럼 구현자 중심의 용어 대신 `박스`, `트레이 사진`, `상품 사진`처럼 사용자가 바로 이해할 수 있는 표현을 사용한다.

### 6.5 일관성과 접근성

버튼, 탭, 검색, 필터, 상태, 오류 메시지는 공통 컴포넌트와 동일한 행동 규칙을 사용한다. 키보드 초점, 긴 상품명, 브라우저 확대와 다양한 화면 크기를 지원한다.

## 7. 정보 구조와 화면

### 7.1 전역 브랜드 선택

모든 주요 화면 상단에 현재 브랜드를 표시한다.

- 브랜드를 바꾸면 상품, 이미지, 검색 범위가 함께 바뀐다.
- 마지막으로 선택한 브랜드를 브라우저에 유지한다.
- 다른 브랜드의 상품은 라벨 선택 목록에 노출하지 않는다.
- 브랜드 생성과 수정은 브랜드 선택 메뉴의 `브랜드 관리`에서 수행한다.

### 7.2 오늘의 작업

복잡한 대시보드 대신 현재 작업량과 다음 행동만 보여준다.

- 라벨이 필요한 트레이 사진 수
- 라벨링을 마친 트레이 사진 수
- 대표 행동: `이어서 라벨링하기`

모델 지표, 통계 그래프, 배포 상태는 표시하지 않는다.

### 7.3 상품 관리

브랜드별 상품과 분류기 학습용 사진을 관리한다.

- 상품 코드와 상품명
- 활성·비활성 상태
- 상품 사진 목록
- 상품 사진 일괄 추가

상품 상세에서 사진을 추가하므로 이미지 종류와 상품은 자동으로 결정한다. 업로드에 성공한 상품 사진은 자동으로 라벨링 완료 상태가 된다.

사용 중인 상품은 삭제하지 않고 비활성화한다. 비활성 상품은 기존 라벨에서 계속 표시하지만 새 박스의 상품 선택 목록에서는 제외한다.

### 7.4 트레이 사진

트레이 사진의 작업함이다.

- `라벨 필요`와 `완료` 탭
- 썸네일과 박스 개수
- 원본 파일명과 업로드 시각
- 파일명 검색
- 상품 필터
- 트레이 사진 일괄 업로드

업로드가 끝나면 `첫 사진 라벨링하기`를 대표 행동으로 제공한다.

### 7.5 라벨링

라벨링 화면은 이미지 캔버스와 선택 박스 편집 영역으로 구성한다. 브랜드·상품 관리 등 다른 기능은 숨겨 박스 작업에 집중하게 한다.

작업 순서는 다음과 같다.

1. 이미지에서 빵을 드래그해 박스를 만든다.
2. 상품을 검색해 박스에 지정한다.
3. 같은 상품을 연속해서 그릴 때 최근 선택 상품을 유지한다.
4. 변경 사항을 자동 저장한다.
5. 모든 박스에 상품이 지정되면 `이 사진 완료`를 활성화한다.
6. 완료하면 다음 미라벨링 이미지로 이동한다.

MVP 단축키는 다음과 같다.

- `N`: 새 박스 그리기
- `Delete`: 선택한 박스 삭제
- `Enter`: 이미지 완료 후 다음 이미지
- `Esc`: 현재 선택 취소

### 7.6 라벨링 기준

데이터 작업자는 다음 기준을 동일하게 적용한다.

1. 판매 단위의 빵 하나당 박스 하나를 생성한다.
2. 빵이 붙어 있거나 겹쳐 있어도 식별 가능한 개별 빵마다 박스를 나눈다.
3. 박스는 화면에 보이는 빵의 외곽에 최대한 밀착하고, 가려진 부분의 크기를 추정해 확장하지 않는다.
4. 빵에 붙어 있는 토핑은 포함하고 트레이, 그림자, 떨어진 부스러기와 포장지는 제외한다.
5. 화면 가장자리에 일부가 잘린 빵도 상품을 식별할 수 있으면 보이는 영역에 박스를 그리고 상품을 지정한다.
6. 상품을 확신할 수 없으면 비슷한 상품으로 추측하지 않는다. 박스의 상품을 지정하지 않고 `라벨 필요` 상태로 남긴다.
7. 화면에서 식별 가능한 모든 빵에 유효한 박스와 상품이 지정되어야 이미지를 완료한다.
8. 빵을 식별하기 어려울 정도로 흐리거나 빵이 없는 트레이 사진은 학습 대상에서 제외한다. MVP에는 별도 보류 상태가 없으므로 해당 이미지를 삭제한다.

박스 크기와 상품 판단이 애매할 때는 처리량보다 일관성을 우선한다. 같은 상황에는 항상 같은 기준을 적용한다.

## 8. 핵심 사용자 흐름

### 8.1 초기 설정과 상품 사진

    브랜드 선택
    → 상품 등록
    → 상품 상세 열기
    → 상품 사진 여러 장 추가
    → 해당 상품의 완료 이미지로 자동 등록

### 8.2 트레이 이미지 라벨링

    트레이 사진 열기
    → 사진 여러 장 업로드
    → 첫 사진 라벨링하기
    → 빵마다 박스 생성
    → 각 박스에 상품 지정
    → 이 사진 완료
    → 다음 미라벨링 사진 자동 이동

### 8.3 작업 재개

    오늘의 작업 열기
    → 라벨이 필요한 사진 수 확인
    → 이어서 라벨링하기
    → 마지막 필터와 브랜드 맥락에서 작업 재개

## 9. 데이터 모델

### 9.1 Brand

- `id`
- `name`
- `status`: `ACTIVE` 또는 `INACTIVE`
- `created_at`
- `updated_at`

브랜드 이름은 활성·비활성 항목을 포함해 중복될 수 없다.

### 9.2 Product

- `id`
- `brand_id`
- `code`
- `name`
- `status`: `ACTIVE` 또는 `INACTIVE`
- `created_at`
- `updated_at`

상품 코드는 같은 브랜드 안에서 중복될 수 없다.

### 9.3 Image

- `id`
- `brand_id`
- `kind`: `PRODUCT` 또는 `TRAY`
- `product_id`: 상품 사진일 때 필수, 트레이 사진일 때 비어 있음
- `storage_key`
- `thumbnail_storage_key`
- `original_filename`
- `mime_type`
- `width`
- `height`
- `byte_size`
- `sha256`
- `labeling_status`: `UNLABELED` 또는 `COMPLETED`
- `revision`: 동시 수정 충돌 감지용 정수
- `created_at`
- `updated_at`

같은 브랜드 안에서 동일한 `sha256`를 가진 활성 이미지는 중복 등록하지 않는다.

### 9.4 BoundingBox

- `id`
- `image_id`
- `product_id`: 라벨링 중에는 비어 있을 수 있으며 이미지 완료 시 필수
- `x1`
- `y1`
- `x2`
- `y2`
- `created_at`
- `updated_at`

좌표는 원본 이미지 기준 0 이상 1 이하의 정규화 값으로 저장한다. `x1 < x2`와 `y1 < y2` 조건을 만족해야 한다. 박스는 `TRAY` 이미지에만 생성할 수 있으며, 박스의 상품은 이미지와 같은 브랜드에 속해야 한다.

## 10. 상태 규칙

### 10.1 상품 사진

- 상품 사진은 활성 상품과 함께 등록해야 한다.
- 정상 등록된 상품 사진은 항상 `COMPLETED`다.
- 다른 상품으로 옮길 때는 같은 브랜드의 활성 상품에만 다시 연결할 수 있다.
- 상품 연결만 제거하는 동작은 제공하지 않는다. 더 이상 필요하지 않은 상품 사진은 삭제한다.

상품 상세에서 추가되는 사진은 현재 상품이 자동 연결되므로 정상 업로드 직후 `COMPLETED`가 된다.

### 10.2 트레이 사진

사용자가 완료를 요청할 때 다음 조건을 모두 만족해야 한다.

- 박스가 한 개 이상 존재한다.
- 모든 박스에 활성 또는 기존에 연결된 비활성 상품이 지정되어 있다.
- 모든 박스 좌표가 유효하다.

완료 후 박스 크기나 이미 지정된 상품을 수정하면 완료 상태를 유지한다. 상품이 없는 박스가 생기거나 모든 박스가 삭제되면 자동으로 `UNLABELED`로 돌아간다.

## 11. 논리 아키텍처

MVP는 하나의 배포 가능한 웹 애플리케이션으로 구성한다.

    Web UI
      ↓
    Application API
      ├─ Relational Database
      └─ Image Storage

- Web UI: 상품 관리, 이미지 탐색, 업로드, 라벨링
- Application API: 검증, 상태 전이, 중복 검사, 동시 수정 제어
- Relational Database: 브랜드, 상품, 이미지 메타데이터, 박스
- Image Storage: 원본과 썸네일

초기 운영 환경은 Windows PC 한 대이며 SQLite와 로컬 파일 저장소를 사용한다. 애플리케이션은 저장소 인터페이스에 의존하며 구체적인 Windows 경로를 업무 로직에서 직접 다루지 않는다. 실행과 저장 구조의 구체적인 결정은 [이미지 관리 MVP 기술 설계](2026-07-15-image-management-technical-design.md)를 따른다.

초기에는 별도 작업 큐나 마이크로서비스를 두지 않는다.

## 12. 데이터 흐름

### 12.1 업로드

    파일 선택
    → 형식·용량·이미지 유효성 검사
    → SHA-256 중복 검사
    → 원본 저장
    → 썸네일 생성과 저장
    → 이미지 메타데이터 저장
    → 파일별 성공·실패 결과 표시

여러 파일 중 일부가 실패해도 성공한 파일은 유지한다. 실패한 파일과 사용자가 취할 수 있는 조치를 함께 표시한다.

### 12.2 박스 자동 저장

    박스 생성·수정·삭제
    → UI에 즉시 반영
    → 마지막 조작 후 500ms가 지나면 revision과 함께 저장
    → 성공하면 저장됨 표시
    → 실패하면 재시도
    → 계속 실패하면 미저장 상태와 이동 경고 표시

서버의 `revision`이 클라이언트가 기준으로 삼은 값과 다르면 저장을 거부한다. 자동 병합으로 작업을 추측하지 않고 최신 내용을 다시 불러오도록 안내한다.

## 13. 오류 처리

### 13.1 업로드

- 지원 형식은 JPEG, PNG, WebP다.
- `.jpg` 또는 `.jpeg`로 제공된 JPEG 기반 MPO는 JPEG 호환 파일로 처리하며, 원본은 보존하고 첫 번째 프레임을 이미지 크기와 썸네일 생성에 사용한다. 세부 규칙은 [JPEG 기반 MPO 업로드 호환 설계](2026-07-20-mpo-jpeg-upload-design.md)를 따른다.
- 파일당 기본 최대 용량은 25MB이며 배포 설정으로 조정할 수 있다.
- 지원하지 않는 형식: 지원 형식 안내
- 손상된 이미지: 해당 파일만 제외
- 최대 용량 초과: 허용 용량 안내
- 중복 이미지: 기존 이미지임을 알리고 건너뜀
- 비활성 상품에 상품 사진 추가: 업로드 시작 전에 차단
- 부분 실패: 성공 결과를 유지하고 실패 파일만 목록화

예시 문구:

> 사진 18장 중 15장을 올렸어요. 3장은 이미 등록된 사진이라 건너뛰었어요.

### 13.2 라벨링

- 상품 미지정 박스를 경고색으로 표시한다.
- 완료할 수 없는 이유를 완료 버튼 근처에 구체적으로 표시한다.
- 저장 상태를 `저장 중`, `저장됨`, `저장하지 못함`으로 표시한다.
- 저장 실패가 지속되면 화면 이동 전에 경고한다.
- 동시 수정 충돌 시 어떤 이미지에서 충돌했는지 알리고 최신 버전 확인을 요구한다.

### 13.3 삭제와 비활성화

- 상품이나 이미지가 연결된 브랜드는 삭제 대신 비활성화한다.
- 상품 사진이나 박스가 연결된 상품은 삭제 대신 비활성화한다.
- 이미지 삭제 전 이미지 수와 함께 삭제되는 박스 수를 표시한다.
- 삭제 확인 후 이미지 메타데이터, 박스, 원본과 썸네일을 함께 제거한다.

## 14. 테스트 전략

### 14.1 단위 테스트

- 상품 사진의 필수 상품 연결과 같은 브랜드 내 재지정
- 트레이 사진 완료 가능 조건
- 미지정 박스 또는 모든 박스 삭제 시 상태 전이
- 박스 좌표 범위와 순서 검증
- 브랜드 간 상품 연결 차단
- 비활성 상품의 신규 지정 차단

### 14.2 API 통합 테스트

- 정상 이미지와 비정상 이미지 업로드
- 동일 브랜드 내 중복 검사
- 일부 파일 업로드 실패
- 브랜드별 상품과 이미지 격리
- 박스 생성, 수정, 삭제
- revision 기반 동시 수정 충돌
- 이미지 삭제 시 연관 데이터 정리

### 14.3 브라우저 전체 흐름 테스트

    브랜드 생성
    → 상품 등록
    → 상품 사진 추가
    → 트레이 사진 업로드
    → 박스와 상품 지정
    → 완료
    → 새로고침 후 결과 유지 확인
    → 검색과 필터 확인

### 14.4 라벨링 상호작용 테스트

- 마우스로 박스 생성, 이동, 크기 조절
- 최소 단축키
- 긴 상품명
- 브라우저 확대와 화면 크기 변화
- 자동 저장 중 네트워크 중단과 복구
- 동시 수정 충돌

### 14.5 사용성 테스트

주 사용자가 별도 구두 설명 없이 실제 업무 데이터로 다음 작업을 수행한다.

1. 상품을 등록하고 상품 사진을 추가한다.
2. 트레이 사진을 올린다.
3. 모든 빵에 박스와 상품을 지정한다.
4. 이미지를 완료하고 다음 이미지로 이동한다.
5. 완료된 이미지를 다시 찾아 수정한다.

사용자가 멈추거나 의미를 묻는 지점, 잘못 선택한 지점, 반복해서 되돌아간 지점을 기록하고 출시 전에 수정한다.

## 15. MVP 완료 기준

- 사용자가 상품 사진과 트레이 사진의 차이를 별도 설명 없이 이해한다.
- 트레이 사진의 모든 빵에 박스와 상품을 지정할 수 있다.
- 새로고침하거나 다음 이미지로 이동해도 저장된 작업이 유실되지 않는다.
- 다른 브랜드의 상품을 잘못 선택할 수 없다.
- 유효하지 않은 트레이 이미지를 완료 처리할 수 없다.
- 업로드와 저장의 부분 실패 후 다음 행동을 알 수 있다.
- 주 사용자가 실제 업무 데이터로 대표 흐름을 확인하고 완료할 수 있다.

## 16. 후속 범위

이미지 관리 MVP가 안정화된 뒤 다음 순서로 확장할 수 있다.

1. 탐지용·분류용 데이터셋 버전과 train/validation/test 분할
2. YOLO 및 분류기 학습 형식 내보내기
3. 학습 실행과 실험 결과 비교
4. 모델 등록, 승인, ONNX 변환
5. 스캐너 단말의 이미지 자동 수집과 모델 배포
6. 사용자 계정, 역할, 작업 할당, 검수와 감사 로그

후속 기능은 이 문서의 MVP 완료 기준을 충족한 뒤 별도의 설계와 구현 계획으로 진행한다.
